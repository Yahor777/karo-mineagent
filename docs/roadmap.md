# Дорожная карта MineAgent Workbench

Единый источник правды по плану развития проекта. Все решения зафиксированы
в обсуждении до старта разработки. Порядок этапов не строгий: 3 и 4 можно
делать параллельно.

## Сквозные принципы (действуют на всех этапах)

1. **Проект строго на русском.** Английский добавляется позже через i18n-слой.
   Инфраструктура i18n ставится сразу на Этапе 0. Идентификаторы кода, JSON-ключи
   и имена моделей остаются английскими.
2. **Не жечь токены.** Prefix-cache-friendly промты, tiering моделей, vision/critic
   только на чекпойнтах, MCP вместо «думанья вслух».
3. **Подтверждение действий пользователем.** Approval Gateway с тремя кнопками
   (Подтвердить / Всегда в сессии / Всегда) — на каждом tool-call и sub-агент вызове.
4. **MCP вместо эмуляции.** ИИ вызывает детерминированные инструменты, не «играет»
   и не угадывает.

---

## Этап 0 — Фундамент (старт)

**Цель:** базовая инфраструктура для всех следующих этапов.

- [ ] Вытащить **все** модели Cloudflare через `/models/search` с нормальным
      парсингом (сейчас логика слабая).
- [ ] Поставить Cloudflare провайдером по умолчанию (`defaultConfig.ts`).
- [ ] **i18n-инфраструктура:** ключи + словарь `ru.ts` (основной), заглушка `en.ts`.
- [ ] **Token-счётчик за сессию** с настраиваемым лимитом (по умолчанию 1М токенов).
      Стоп-предложение появляется **только после завершения ответа модели**, если
      лимит превышен. Кнопка «больше не показывать в сессии».
- [ ] Редактируемый базовый системный промт.

## Этап 1 — Approval Gateway + Sub-агенты (каркас)

**Цель:** пользователь контролирует каждое действие; main-модель может вызывать
sub-агентов по запросу.

- [ ] **Approval Gateway:** 3 кнопки на каждый tool-call.
      Per-tool и per-sub-агент (чтобы не было спама модалок).
      `approvalMode` из конфига = начальное состояние слоя.
- [ ] **Тумблер sub-агентов** глобально включить/выключить.
- [ ] **Каркас sub-агентов:** конфиг-модель + UI.
      Поля: id (уникальный), отображаемое имя, модель, специальность, промт
      (базовый + надстройка), права/инструменты, режим памяти (4 опции).
- [ ] Main = «разнорабочий». Делегирует **только** при:
      vision, ревизии, превышении итераций, явном запросе пользователя.

## Этап 2 — Code-MCP: tool-calling + build/diagnose loop

**Цель:** замкнутый цикл кода «написал → собрал → починил».

- [ ] Function-calling для `repo.read`, `repo.patch`, `gradle.run`.
- [ ] Автоматический `gradle.build` после `repo.patch`.
- [ ] Если `exitCode !== 0` → `logParser` → кормить ошибку обратно в модель
      (макс. 2 итерации).

## Этап 3 — Blockbench MCP-клиент

**Цель:** ИИ строит и правит 3D-модели и анимации в живом Blockbench.

- [ ] Проверить LICENSE `jasonjgardner/blockbench-mcp-plugin`.
      MIT/Apache → fork. GPL/copyleft → свой.
- [ ] Найти дыры и слабые места в существующем MCP, прокачать.
- [ ] MCP-клиент в MineAgent: подключение к `localhost:3000/bb-mcp`.
- [ ] Прокси tool-calls от модели в живой Blockbench (модель, кости, keyframes).
- [ ] Скриншот-рендер модели из Blockbench → vision-оценка.

## Этап 4 — Minecraft Dev Bridge (MCP-сервер в моде)

**Цель:** ИИ видит игру и проверяет эффекты/механики детерминированно.

- [ ] Мод-модуль `mineagent-bridge`, MCP-совместимый endpoint.
      Активен **только** в dev-окружении (`isDevelopmentEnvironment()`).
- [ ] Инструменты: `summon`, `apply_effect`, `set_camera`, `screenshot → base64`,
      `get_state`, `reload_resources`.
- [ ] Полностью отключается в production-сборке.
- [ ] Extension как MCP-клиент к мосту.

## Этап 5 — Vision + Critic loop

**Цель:** две модели оценивают артефакты (скрины из игры / рендеры Blockbench).

- [ ] Vision-модель (kimi-k2.7 / llama-4-scout) оценивает: «модель видна? эффект
      выглядит как задумано? анимация проигрывается?».
- [ ] **Critic по умолчанию = другая модель.** Self-critique возможен с
      предупреждением в UI (объективность ниже).
- [ ] Critic получает компактный артефакт (project map + задача + объект оценки),
      **без хода мыслей main** (убирает anchoring).
- [ ] Консенсус = применить. Разногласие = спросить пользователя.
- [ ] Triggered на чекпойнтах дизайна, не на каждой строке кода.

## Этап 6 — JJK-скиллы + reference packs

**Цель:** специализированный workflow для мода по JJK.

- [ ] Reference Pack: оригинальная Minecraft-механика, без копирования лора/имён.
- [ ] Скиллы в `.mineagent/skills/`: боевые эффекты, арена домена и т.д.
- [ ] Source Ledger по категориям (персонажи / лор / оружие / локации) с тегами.

## Сквозная фишка — Source Ledger (вкладка ссылок)

Расширение существующего `ResearchLedger`. Действует поверх всех этапов.

- [ ] **Категоризация:** модель предлагает категорию при сохранении, пользователь
      может переносить вручную.
- [ ] **RAG-индекс:** модель решает КАКИЕ ссылки релевантны текущей задаче, читает
      только их. Без этого — пожар токенов.
- [ ] **Web-поиск** через MCP/модель с доступом в интернет.
- [ ] **Полный файл заметок** отдельно от UI-вкладки (пользователь редактирует оба).
- [ ] **`source: user` всегда побеждает** над `source: model` при retrieval.
- [ ] **Кнопка «отправить изменения модели» убрана** как бессмысленная: твои правки
      просто приоритетнее в retrieval автоматически.

## Режимы памяти sub-агента (4 опции)

| Режим | Поведение |
|---|---|
| Без памяти (stateless) | по умолчанию для critic |
| Память в рамках задачи | помнит до конца твоей задачи, потом чистится |
| Память в рамках сессии | помнит до рестарта VS Code |
| Спрашивать каждый раз | модал после каждой итерации |

## Approval Gateway — 3 кнопки

| Кнопка | Поведение |
|---|---|
| Подтвердить один раз | выполнить эту команду, спросить снова в следующий раз |
| Всегда в этой сессии | больше не спрашивать этот конкретный tool до перезапуска VS Code |
| Всегда | записать в `config.json` → `autoApproveTools[]`, с возможностью отозвать |

## Tiering моделей (пользователь выбирает)

- Рутина (чтение/мелкие патчи) → лёгкая модель.
- Сложные решения → frontier-модель.
- Vision и critic — только на чекпойнтах.
- Каждый sub-агент = своя модель + специальность + промт + права.
