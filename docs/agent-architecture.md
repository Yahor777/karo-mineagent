# Архитектура агента MineAgent

Технические спецификации всех решений, зафиксированных в обсуждении.
Сопутствующий документ: `roadmap.md` (порядок этапов).

## Топология

```
                    ┌─────────────────────────────────┐
                    │   MineAgent (VS Code extension)  │

                    │   │   Approval Gateway    │ ← 3 кнопки подтверждения
                    │   │   (ask / session /    │      на каждый tool-call
                    │   │    always)            │      и sub-агент вызов
```

## 1. Main Agent (разнорабочий)

Основной исполнитель. Делает всю работу сам по умолчанию.

**Делегирует sub-агенту ТОЛЬКО когда:**
1. Пользователь явно попросил.
2. Задача вне компетенции main (vision, ревизия дизайна).
3. Превышен лимит итераций main.
4. Тумблер sub-агентов включён И есть approval на конкретный вызов.

Без approval gateway на вызов sub-агента — вызов невозможен.

## 2. Sub-агенты

### Конфигурация sub-агента

```typescript
interface SubAgentConfig {
  id: string;              // уникальный (например: "reviewer-jjk")
  displayName: string;     // для UI (можно дубликаты)
  model: string;           // id модели из реестра провайдера
  specialty: string;       // "reviewer" | "researcher" | "vision" | "custom"
  promptOverride?: string; // надстройка над базовым промтом (опционально)
  allowedTools: string[];  // whitelist tool names
  memoryMode: MemoryMode;  // см. ниже
  enabled: boolean;        // тумблер глобально
}
```

### Специальность = пресет-бандл

Специальность это не ярлык, а набор `{промт + инструменты + права}`:

| Специальность | Промт | Инструменты по умолчанию |
|---|---|---|
| `reviewer` | ревизия дизайна/кода | только чтение (без `repo.patch`) |
| `researcher` | web-поиск, заполнение Source Ledger | `web.search`, `reference.*` |
| `vision` | оценка скринов/рендеров | `minecraft.screenshot`, `blockbench.render` |
| `custom` | полностью пользовательский | юзер задаёт вручную |

Юзер может отвязать пресет и настроить суб-агента с нуля.

### Режимы памяти

```typescript
type MemoryMode =
  | "none"          // stateless — по умолчанию для critic
  | "task"          // помнит до конца текущей задачи
  | "session"       // помнит до рестарта VS Code
  | "ask";          // спрашивать модалом после каждой итерации
```

### Жизненный цикл sub-агента

- Вызов main → approval gateway (3 кнопки) → запуск sub.
- Таймаут 60с. Если sub завис → main получает «сорри, sub-агент отвалился».
- Если sub ничего не сделал — main продолжает сам.
- Max-tokens на вызов: пока не настраиваем (later).

## 3. Critic (особый sub-агент для оценки)

### Правила

1. **По умолчанию critic = ДРУГАЯ модель** (не та же, что main).
2. Self-critique (та же модель) возможен, но с **предупреждением в UI**:
   «объективность ниже — у модели те же слепые зоны тренировки».
3. Critic получает **компактный артефакт**, не ход мыслей main:

```typescript
interface CriticArtifact {
  projectMap: ProjectMap;   // компактный JSON карты проекта
  taskDescription: string;  // что оцениваем (одно предложение)
  artifact: string;         // сам объект оценки: код / скрин base64 / geo.json
  mode: "code" | "design" | "vision";
}
```

Убирает anchoring bias — critic не «слышит» аргументацию main.

4. **Triggered на чекпойнтах дизайна**, не на каждой строке кода.
5. Консенсус = применить. Разногласие = спросить юзера.

## 4. Approval Gateway

3 кнопки на каждый tool-call и sub-агент вызов:

| Кнопка | Эффект | Хранилище |
|---|---|---|
| Подтвердить один раз | выполнить, спросить снова в следующий раз | in-memory |
| Всегда в этой сессии | не спрашивать этот tool до рестарта | in-memory session set |
| Всегда | не спрашивать никогда (отзываемо) | `config.json → autoApproveTools[]` |

### Per-tool и per-sub-агент

Одобрение действует на конкретный tool (`repo.patch`) или конкретного sub-агента
(`reviewer-jjk`), а не на «все tool-calls». Защищает от модалко-спама когда
main зовёт 5 sub-агентов подряд.

### `approvalMode` из конфига = начальное состояние

```typescript
type ApprovalMode = "ask" | "workspace" | "auto-readonly";
```

- `ask` — подтверждение на всё (по умолчанию).
- `workspace` — читать без подтверждения, писать/запускать — с подтверждением.
- `auto-readonly` — все read-only tools без подтверждения, остальное с подтверждением.

Кнопки gateway дают юзеру override этого режима в рантайме.

## 5. Token Budget

### Счётчик за сессию

```typescript
interface TokenBudget {
  sessionUsed: number;     // потрачено с момента старта VS Code
  sessionLimit: number;    // настраиваемый, по умолчанию 1_000_000
  inputTokens: number;
  outputTokens: number;
  visionCalls: number;
}
```

### Правила стоп-предложения

- **НЕ прерывает** выполняющийся запрос модели (модель продолжает до конца ответа).
- **Появляется только ПОСЛЕ завершения ответа модели**, если `sessionUsed > sessionLimit`.
- Модал: «Превышен лимит N токенов за сессию. Продолжить?»
  - Продолжить → продолжаем
  - Остановить → стоп
  - Больше не показывать в этой сессии → скрыть до рестарта
- Лимит настраивается в настройках.

### Токен-экономия (принципы)

- **Prefix-cache-friendly:** system prompt + project map идентичны между вызовами.
- **Tiering моделей:** рутина → лёгкая, решения → frontier, vision/critic → только
  на чекпойнтах.
- **MCP вместо «думанья»:** модель вызывает `repo.read`, не угадывает.
- **Critic не дублирует:** получает только артефакт + одно предложение, не весь контекст.

## 6. System Prompt

### Структура: базовый + надстройка

```
[БАЗОВЫЙ ПРОМТ — общий, редактируемый юзером]
  - роль MineAgent
  - правила работы (Minecraft API, цитирование, безопасность)
  - формат ответов

[+ НАДСТРОЙКА sub-агента — если это sub-агент]
  - специализация (reviewer/researcher/vision/custom)
  - разрешённые инструменты
  - ограничения прав

[+ КОНТЕКСТ ВЫЗОВА]
  - project map (compact JSON)
  - задача
  - артефакт (для critic)
```

Базовый промт хранится редактируемым в `.mineagent/`. У каждого sub-агента своя
надстройка поверх. Дефолтные sub-агенты = пресеты.

## 7. Source Ledger (вкладка ссылок)

Расширение существующего `ResearchLedger` в `configService.ts`.

### Структура

```typescript
interface SourceEntry {
  url: string;
  title?: string;
  category: SourceCategory;  // персонажи | лор | оружие | локации | misc
  tags: string[];            // перекрёстные ссылки (оружие персонажа X)
  summary: string;           // краткое описание что нашли
  fullNotes?: string;        // полные заметки (модель пишет при поиске)
  source: "user" | "model";  // ВАЖНО: user всегда побеждает
  status: "candidate" | "accepted" | "rejected";
  addedAt: string;
}

type SourceCategory = "characters" | "lore" | "weapons" | "locations" | "misc";
```

### RAG-индекс (критично для токенов)

Модель НЕ читает весь Ledger каждый раз. Вместо этого:

1. Перед вызовом модели — embedding/ключевые слова по текущей задаче.
2. Retrieval: выбираются только релевантные записи (top-K).
3. В контекст модели попадают только они.

Без этого — раздувание контекста, нарушение «не жечь токены».

### Web-поиск

- Через MCP web-search сервер ИЛИ модель с доступом в интернет.
- Модель ищет → пишет full notes в отдельный файл → добавляет entry в Ledger.
- Пользователь редактирует и вкладку, и файл.
- Кнопка «отправить изменения модели» **убрана** — твои правки (`source: user`)
  автоматически приоритетнее в retrieval.
- Категоризацию предлагает модель при сохранении; юзер может перенести.

## 8. Внешние MCP-источники

MineAgent — MCP-клиент к трём источникам:

| Источник | Назначение | Активация |
|---|---|---|
| Code tools (внутренние) | repo.read/patch, gradle.run | всегда |
| Minecraft Dev Bridge | summon, effect, camera, screenshot | только в dev-сборке мода |
| Blockbench MCP | модель, кости, keyframes, анимации | когда запущен Blockbench |

Все три — через approval gateway.

## 9. i18n-инфраструктура

Проект строго на русском. Английский добавляется позже через тот же слой.

### Принципы

- Все **UI-строки и тексты для модели** через i18n-ключи.
- **Идентификаторы кода, JSON-ключи, имена моделей — остаются английскими** (стандарт).
- Словарь: `src/i18n/ru.ts` (основной), `src/i18n/en.ts` (заглушка на старте).
- Загрузка словаря по настройке `mineagent.language` (позже).

Это дёшево на старте и сэкономит недели при добавлении английского.
