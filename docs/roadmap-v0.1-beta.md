# MineAgent — План до v0.1 beta

Согласовано с пользователем. Главные принципы:
- ВЫБОР МОДЕЛИ СВЯЩЕНЕН. Никакой тихой смены модели. Повтор на той же модели при сетевом
  сбое — авто; смена модели (model-not-found / нет capability / контекст не влез) — только
  с подтверждением пользователя.
- Память = 3 слоя: быстрый структурный индекс + ИИ-индекс (по кнопке, выбор модели) + RAG.
  ИИ-индекс кормит и память, и RAG. Инкрементальный кэш.
- Веб-поиск: реальный (Firecrawl) + бесплатный DuckDuckGo-фоллбэк, который РЕАЛЬНО помогает.
- Минусы каждого провайдера нивелируем единым резолвером capability + preflight-пробой.
- effort (reasoning effort) для моделей, которые поддерживают.
- UI — глобальная переработка. Слэш "/" как палитра всех функций.

## Фаза 1 — Надёжные провайдеры (безопасность, фундамент)
Файлы: providers/ProviderAdapter.ts, openaiCompatibleProvider.ts, anthropicProvider.ts,
providerRegistry.ts, orchestrator.ts (askConfiguredModel/selectModelCandidates), config/types.ts.
- [P1.1] Убрать тихий фоллбэк. Ошибки делим: transient (network/429/5xx) → retry той же модели
  с backoff; hard (model-not-found/нет capability/контекст) → СТОП + событие в UI с выбором
  «повторить / выбрать другую модель / отменить». Никогда не выбирать модель за пользователя.
- [P1.2] Замок модели: config.providers.lockModel — отключает авто-тиринг; в UI всегда видно,
  какая модель отвечала.
- [P1.3] Capability-резолвер: под задачу (tools/vision/json/контекст) фильтруем модели; если
  выбранная не тянет — спрашиваем, не подменяем. Vision к не-vision модели больше не «молчит».
- [P1.4] Preflight-проба при выборе модели: крошечный запрос, кэшируем РЕАЛЬНЫЕ возможности,
  помечаем нерабочие (нивелирует «врущий» /models: qwen3-coder-next-fp8, glm без vision).
- [P1.5] effort: добавить reasoning_effort (low/medium/high) в ChatRequest + проброс в
  openaiCompatible; capability-флаг effortLevels; гейтить только для reasoning-моделей.
- [P1.6] Anthropic: реализовать tools (Messages API tools/tool_use/tool_result) и vision
  (base64 image blocks). До готовности — честно не предлагать Claude для tools/vision.
Acceptance: на kimchi tool-loop/vision стабильны; смена модели только через подтверждение;
qwen/glm не дают тихих пустых ответов; Claude умеет tools ИЛИ явно помечен chat-only.

## Фаза 2 — Память: 3 слоя + Web-RAG
Файлы: repo/repoIndexer.ts (есть), новые memory/workspaceIndex.ts, providers/localEmbedding*.ts,
knowledge/*, webview/* (кнопка), orchestrator.ts, tools/*.
- [P2.1] ИИ-индекс (отдельный слой): кнопка «Проиндексировать с ИИ» → выбор провайдера+модели
  (модель индекса ≠ модель работы) → конспект на файл/класс (что делает, что регистрирует,
  зависимости, точки расширения) + общая карта архитектуры. Хранить в .mineagent/workspace-index/.
  Инкрементально по хэшу файла. Прогресс/возраст индекса в UI.
- [P2.2] Локальные embeddings (transformers.js/fastembed, bge-m3, оффлайн) → семантический RAG
  без зависимости от провайдера. Конспекты ИИ-индекса и записи базы знаний индексируются.
- [P2.3] memory.note tool: агент сам пишет решения/находки в project.md и базу знаний во время работы.
- [P2.4] Web-RAG реальный: заменить DuckDuckGo Instant Answer на реальный поиск через Firecrawl
  (подключён в Gumloop) — поиск + скрейп страницы → конспект «что нашёл» → запись url+notes+теги.
- [P2.5] Бесплатный режим: DuckDuckGo HTML-результаты (не Instant Answer) + лёгкий скрейп, чтобы
  реально находил. Переключатель «экономный/полный» поиск.
Acceptance: агент находит «где обрабатывается X» через RAG, не читая весь репозиторий; веб-поиск
возвращает реальные ссылки с непустыми конспектами; ИИ-индекс инкрементальный и кэшируется.

## Фаза 4 — UI: глобальная переработка
Файлы: webview/html.ts, media/mineagent.css, media/mineagent.js, MineAgentWebviewProvider.ts.
- [P4.1] Новый каркас: чёткий layout, тёмная тема, состояния загрузки/ошибки/пусто, гуманизация ошибок.
- [P4.2] Слэш "/" — палитра всех функций в поле ввода: режимы (ask/plan/build/playtest), инструменты,
  индекс/ИИ-индекс, поиск знаний, git-операции, выбор модели/effort, память. Автодополнение.
- [P4.3] Селекторы модели и effort прямо в инпуте; индикатор «какая модель отвечает» + замок модели.
- [P4.4] Панели: Память (project.md), База знаний, Git, Индекс (структурный + ИИ, возраст/прогресс).
- [P4.5] Подтверждения смены модели/опасных действий — нормальные модалки, а не тихие фейлы.
Acceptance: интерфейс не «убогий»: тёмная тема, слэш-палитра, видимый выбор модели+effort, панели
памяти/знаний/git/индекса, явные подтверждения.

## Порядок
1 → 2 → 4(каркас) параллельно с 3 → 5. Сначала безопасность провайдеров (Фаза 1), затем память/RAG
(Фаза 2) как сердце, UI-каркас (Фаза 4) подтягиваем рано, чтобы было видно фичи; git и релиз — в конце.
