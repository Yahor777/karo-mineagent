# Фаза 1 — Память проекта MineAgent («не забывать ничего»)

Это полная продумка того, как память должна работать и помогать. Цель пользователя —
продолжать проект сколь угодно долго, ничего не забывая, и чтобы это работало с
**любым** провайдером (kimchi/custom, Cloudflare, Fireworks, WaveSpeed, OpenAI,
Anthropic), а не только там, где есть embeddings.

## 1. Принцип: три слоя памяти

У агента память делится на три слоя по времени жизни и стоимости:

1. **Рабочая память (контекст запроса)** — то, что уходит в модель прямо сейчас:
   системный промпт + карта проекта (ProjectMap) + извлечённые скиллы/знания.
   Живёт один запрос. Дорогая (токены), ограничена окном модели.

2. **Долговременная память проекта** — `.mineagent/project.md`. Это сделано в Фазе 1.
   Один человекочитаемый файл, который агент ведёт сам и подмешивает в НАЧАЛО
   контекста при каждом запросе. Провайдеро-независим (просто текст, без embeddings).
   Хранит факты, которые нельзя терять между сессиями: идентичность мода,
   конвенции, добавленный контент, принятые решения, открытые вопросы, журнал задач.

3. **Эпизодическая память (сессии)** — `.mineagent/sessions/*.json`. Уже существует
   (sessionService). Полная история чатов. Со временем разрастается и переполняет
   контекст — поэтому в Фазе 1 (п.6) предусмотрена суммаризация старых сессий
   в долговременную память.

Ключевая идея: **факт, который важен дольше одной сессии, должен «всплыть» из
эпизодической памяти в долговременную** (project.md), иначе он будет потерян при
сжатии истории. Долговременная память — это то, что делает агента «помнящим».

## 2. Что сделано в Фазе 1 (этот PR)

### 2.1 `ProjectMemoryService` (`src/memory/projectMemory.ts`)
Сервис над `.mineagent/project.md`. Файл размечен HTML-комментариями на разделы,
что даёт три свойства: человекочитаемость, машинная адресуемость (меняем один
раздел, не трогая остальные и ручные правки), идемпотентность (повторный sync не
плодит дубликаты).

Разделы:
- `auto:identity` — факты из ProjectMap (загрузчик, версия MC, Java, modId,
  число реестров/обработчиков). Перезаписывается агентом при каждом запуске.
- `section:conventions` — конвенции регистрации/архитектуры. Агент + пользователь.
- `section:content` — добавленный в мод контент. Агент + пользователь.
- `section:decisions` — принятые решения и причины. Агент + пользователь.
- `section:open` — открытые вопросы / TODO. Агент + пользователь.
- `auto:log` — журнал задач (последние N=20). Пишется автоматически после каждого run.

API: `ensure`, `readRaw`, `syncIdentity`, `appendToSection`, `appendRunLog`,
`renderForPrompt`. Дедупликация в `appendToSection` позволяет агенту безопасно
вызывать запись после каждой задачи без риска дублей.

### 2.2 Интеграция в оркестратор (`src/orchestrator/orchestrator.ts`)
- Конструктор принимает опциональный `ProjectMemoryService` (последним параметром).
- В начале `run()`: после построения ProjectMap агент `syncIdentity(...)` (помнит
  базовые факты даже если индексатор потом недоступен) и `renderForPrompt()` →
  кладёт в `request.projectMemory`.
- В `createChatMessages`: блок памяти идёт **первым** в пользовательском сообщении —
  модель опирается на него прежде всего.
- В конце `run()`: `appendRunLog(...)` сохраняет «что делали и чем закончилось».

### 2.3 Интеграция в UI-слой (`src/webview/MineAgentWebviewProvider.ts`)
`ProjectMemoryService` создаётся рядом с `SessionService` (корень воркспейса) и
передаётся в оркестратор. Память включена всегда и не зависит от настроек RAG
(`knowledgeTopK`/`skillsTopK` по умолчанию 0 — то есть embeddings-память выключена,
а текстовая память проекта работает).

### 2.4 Почему это решает задачу пользователя
- Работает с kimchi (`custom`) и любым другим провайдером — это просто текст.
- Переживает перезапуски VS Code и смену сессии (файл на диске в воркспейсе).
- Переживает сжатие истории чата — факты лежат отдельно от сессий.
- Прозрачно: пользователь открывает `project.md`, видит и правит память руками.

## 3. Что дальше в Фазе 1 (следующие PR, по приоритету)

### 3.1 Инструмент памяти для модели (`memory.note`) — высокий приоритет
Сейчас разделы conventions/content/decisions/open наполняются программно. Чтобы
агент **сам** записывал решения по ходу tool-loop, добавить tool `memory.note`
(аргументы: `section`, `text`) в `toolSchemas` + обработчик в `toolDispatcher`,
который зовёт `appendToSection`. Это превратит память из пассивной в активную:
агент после каждой значимой задачи фиксирует «добавил предмет X», «договорились
делать Y». Контракт уже подготовлен сервисом — нужен только tool + dispatch.

### 3.2 Команда и UI-вкладка «Память проекта»
- Команда `MineAgent: Open Project Memory` (открыть `project.md` в редакторе).
- Вкладка в webview: показывать разделы, давать кнопку «забыть запись».

### 3.3 Локальные эмбеддинги — провайдеро-независимый семантический RAG
Сейчас `EmbeddingService` зависит от `provider.embeddings`, которого у kimchi нет —
поэтому Knowledge Base/Skills retrieval по умолчанию выключены. Чтобы включить
семантический поиск независимо от провайдера, добавить `LocalEmbeddingService`,
реализующий тот же контракт `embed/embedBatch`, но через локальную модель в Node:
- `@xenova/transformers` (transformers.js) или `fastembed` с моделью
  `bge-small`/`bge-m3` (multilingual — важно для русского).
- Модель скачивается один раз и работает оффлайн, без API-ключей и токенов.
- Вектора кэшируются рядом с записями (как уже сделано в knowledgeBase).
Тогда `knowledgeTopK`/`skillsTopK` можно поднять с 0, и память станет ещё и
семантической, оставаясь провайдеро-независимой. project.md при этом — источник
фактов, а локальный RAG — индекс по большому объёму (доки, прошлые решения).

### 3.4 Авто-карта проекта в начале диалога
ProjectMap уже строится и уходит в контекст. Добавить компактный «оглавление
репозитория» (ключевые классы/пакеты) в память, чтобы агент всегда «помнил»
структуру даже при первом сообщении новой сессии.

### 3.5 Суммаризация старой истории (sessions → memory-notes)
Когда сессия становится длинной, контекст переполняется. Решение:
- При превышении порога (например > N сообщений или > M токенов) суммаризировать
  самые старые сообщения в краткие «заметки памяти» и переносить устойчивые факты
  в `project.md` (decisions/content), а сырые сообщения сворачивать.
- Так контекст не растёт бесконечно, но факты сохраняются навсегда.
- Суммаризацию делает текущая модель (провайдеро-независимо), запускается фоном
  после run.

## 4. Принципы записи в память (чтобы она помогала, а не мешала)

1. **Факты, а не пересказ.** В память идут устойчивые решения и сущности
   («mod id = kimijjk», «регистрация через DeferredRegister», «добавлен предмет
   cursed_charm»), а не диалог целиком.
2. **Дедупликация.** Повтор той же записи игнорируется — агент может писать после
   каждой задачи без замусоривания.
3. **Приоритет пользователя.** Ручные правки в разделах сохраняются; auto-разделы
   (identity, log) — зона агента. Записи пользователя помечаются.
4. **Ограничение объёма.** Журнал хранит последние N, рендер в промпт обрезается
   по `maxChars` — защита от пожара токенов.
5. **Память — первой в контексте.** Модель должна опираться на принятые решения и
   не противоречить им.

## 5. Как проверить вручную
1. Открыть мод-воркспейс, задать любой запрос агенту.
2. Появится `.mineagent/project.md` с заполненной идентичностью и записью журнала.
3. Задать второй запрос — в начале контекста модель уже «помнит» проект; журнал
   пополнится. Файл можно открыть и править руками — правки сохранятся.
